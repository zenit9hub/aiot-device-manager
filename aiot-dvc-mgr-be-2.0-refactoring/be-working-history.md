## 🏗️ 아키텍처 패턴 결정

### 검토 옵션

#### Option 1: Clean Architecture + DDD ⭐
```
src/
├── application/      # 유스케이스
├── domain/           # 엔티티
├── infrastructure/   # 구현체
└── presentation/     # API
```

**장점**:
- 의존성 역전 원칙
- Firebase/MySQL 교체 가능
- 테스트 용이

**단점**:
- 초기 설정 복잡

#### Option 2: Layered Architecture
```
src/
├── controllers/
├── services/
├── repositories/
└── models/
```

**장점**:
- 간단하고 직관적

**단점**:
- 의존성 관리 어려움

#### Option 3: Hexagonal Architecture
```
src/
├── core/
├── ports/
└── adapters/
```

**장점**:
- 비즈니스 로직 독립성

**단점**:
- 러닝 커브
